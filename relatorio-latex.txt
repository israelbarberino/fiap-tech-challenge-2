\begin{resumo}
Este relatório técnico apresenta o projeto, a arquitetura e a implementação de uma API REST voltada ao gerenciamento de usuários, desenvolvida no contexto do Tech Challenge -- Fase 1 do curso de Pós-Graduação em Arquitetura e Desenvolvimento Java da FIAP. A solução foi construída utilizando Java 17 e Spring Boot 3.x, fundamentada em princípios consolidados da engenharia de software, com destaque para a Clean Architecture e os princípios SOLID.

O sistema contempla operações completas de criação, consulta, atualização e remoção de usuários, bem como validação de credenciais de acesso, adotando práticas de segurança, validação em múltiplas camadas, documentação automatizada e execução reprodutível por meio de Docker Compose. O relatório enfatiza as decisões arquiteturais, os trade-offs envolvidos e os impactos dessas escolhas na manutenibilidade, testabilidade e evolução futura do sistema.

\textbf{Palavras-chave}: API REST. Clean Architecture. Spring Boot. Engenharia de Software. Docker.
\end{resumo}

% =====================
% ABSTRACT
% =====================
\begin{abstract}
This technical report presents the design, architecture, and implementation of a REST API for user management, developed as part of the Tech Challenge -- Phase 1 of the Graduate Program in Java Architecture and Development at FIAP. The solution was built using Java 17 and Spring Boot 3.x, grounded in established software engineering principles, particularly Clean Architecture and SOLID principles.

The system provides full CRUD operations, credential validation, multi-layer validation mechanisms, automated API documentation, and reproducible execution through Docker Compose. This report focuses not only on functional implementation but also on architectural decisions, trade-offs, and their impact on maintainability, testability, and future system evolution.

\textbf{Keywords}: REST API. Clean Architecture. Spring Boot. Software Engineering.
\end{abstract}

\chapter{Introdução}

O desenvolvimento de sistemas distribuídos modernos tem ampliado significativamente a relevância de APIs REST como elemento central de integração entre aplicações. Em ambientes corporativos, decisões arquiteturais inadequadas em serviços backend tendem a gerar sistemas de difícil manutenção, alto acoplamento e elevada dívida técnica ao longo do tempo.

Nesse contexto, este trabalho aborda o problema arquitetural da construção de uma API REST que seja, simultaneamente, funcional, segura e arquiteturalmente sustentável. O foco não se restringe à entrega de endpoints, mas à adoção consciente de princípios arquiteturais que favoreçam a evolução do sistema.

\section{Objetivo Geral}

Projetar e implementar uma API REST para gerenciamento de usuários, fundamentada em princípios consolidados de arquitetura de software, garantindo manutenibilidade, testabilidade e clareza estrutural.

\section{Objetivos Específicos}

\begin{itemize}
  \item Aplicar os princípios da Clean Architecture no desenvolvimento de uma API REST.
  \item Modelar entidades e relacionamentos de forma consistente com o domínio do problema.
  \item Projetar endpoints REST seguindo boas práticas de design e semântica HTTP.
  \item Implementar validações e mecanismos de segurança adequados.
  \item Disponibilizar documentação automatizada e ambiente de execução reprodutível.
\end{itemize}

  \item \textbf{Documentação Contínua}: Atualização de documentação (Swagger, README, ARCHITECTURE) conforme evolução do código.
  
  \item \textbf{Validação Final}: Verificação de conformidade com requisitos, execução completa de testes e validação de cobertura.

\chapter{Arquitetura da Aplicação}

A arquitetura da aplicação foi concebida com base nos princípios da Clean Architecture, conforme proposto por Martin (2017). Essa abordagem promove a separação entre regras de negócio e detalhes de infraestrutura, favorecendo a independência de frameworks e a testabilidade do sistema.

A aplicação é estruturada em camadas bem definidas: apresentação, serviços, domínio, persistência e infraestrutura. O fluxo de dependências ocorre sempre das camadas externas para as internas, respeitando o princípio da inversão de dependência.

\subsection{Camadas da Aplicação}

\begin{enumerate}
  \item \textbf{REST API Layer}: Controladores (`AuthController`, `UserController`) que expõem endpoints HTTP e gerenciam requisições/respostas.
  \item \textbf{Business Logic Layer}: Serviço (`UserService`) que concentra toda a lógica de negócio e coordenação entre camadas.
  \item \textbf{Data Access Layer}: Repositório JPA (`UserRepository`) que abstrai operações de persistência.
  \item \textbf{Domain Layer}: Entidades (`User`, `Address`), enums (`UserRole`) e objetos de valor que representam o domínio.
  \item \textbf{Infrastructure}: Configurações de segurança, documentação da API, banco de dados e tratamento de exceções.
\end{enumerate}

\section{Análise de Alternativas Arquiteturais}

Arquiteturas em camadas tradicionais foram consideradas, porém rejeitadas devido ao acoplamento excessivo entre regras de negócio e frameworks. A Clean Architecture foi escolhida por oferecer maior flexibilidade, melhor isolamento do domínio e facilidade de evolução futura.

\section{Padrões de Design Implementados}

A solução utiliza diversos padrões de projeto consolidados:

\begin{itemize}
  \item \textbf{Repository Pattern}: Abstração do acesso a dados através de `UserRepository`.
  \item \textbf{Service Layer}: Concentração da lógica de negócio em `UserService`.
  \item \textbf{DTO Pattern}: Transferência de dados através de `UserCreateRequest`, `UserUpdateRequest`, `UserResponse`, etc.
  \item \textbf{Value Object}: Modelagem de `Address` como objeto de valor embutido na entidade `User`.
  \item \textbf{Factory Pattern}: Construção de `UserResponse` a partir da entidade `User`.
  \item \textbf{Dependency Injection}: Injeção de dependências via construtor (Spring Framework).
  \item \textbf{Exception Handling}: Tratamento centralizado de exceções com `GlobalExceptionHandler` conforme RFC 7807.
\end{itemize}

\chapter{Modelagem de Dados}

O domínio do sistema é centrado na entidade \textit{User}, responsável por representar usuários do sistema. A entidade possuí os seguintes atributos principais:

\begin{itemize}
  \item \textbf{id}: Identificador único (UUID).
  \item \textbf{name}: Nome do usuário (obrigatório, 3-100 caracteres).
  \item \textbf{email}: Endereço de correio eletrônico (obrigatório, único, validado).
  \item \textbf{login}: Nome de acesso (obrigatório, único, 3-50 caracteres).

\section{Camada de Domínio}

A entidade \texttt{User} é o núcleo do domínio, representando um usuário do sistema. Implementa validações via Jakarta Validation:

\begin{itemize}
  \item \textbf{@NotBlank}: Campos obrigatórios (name, email, login).
  \item \textbf{@Email}: Validação de formato de email.
  \item \textbf{@Size}: Restrições de tamanho (3-100 caracteres para nome).
  \item \textbf{@Unique}: Constraint customizada para unicidade de email e login.
  \item \textbf{@CreationTimestamp}: Data de criação automática.
  \item \textbf{@UpdateTimestamp}: Data de atualização automática.
\end{itemize}

O endereço é modelado como \texttt{@Embedded}, permitindo tratá-lo como um objeto coeso sem complexidade de tabelas separadas.

\section{Camada de Persistência}

\texttt{UserRepository} estende \texttt{JpaRepository}, fornecendo operações CRUD padrão mais três métodos customizados:

\begin{lstlisting}[language=java, caption=Métodos Customizados do Repository]
Optional<User> findByEmail(String email);
Optional<User> findByLogin(String login);
List<User> findByNameContainingIgnoreCase(String name);
\end{lstlisting}

Esses métodos são derivados automaticamente pelo Spring Data JPA a partir do nome, sem necessidade de SQL manual.

\section{Camada de Serviço}

\texttt{UserService} concentra toda a lógica de negócio:

\begin{itemize}
  \item \textbf{createUser()}: Cria novo usuário com validações e hash de senha.
  \item \textbf{getUserById()}: Recupera usuário por ID ou lança exceção.
  \item \textbf{getUsersByName()}: Busca por nome com case-insensitive.
  \item \textbf{updateUser()}: Atualiza dados sem modificar senha.
  \item \textbf{changePassword()}: Altera senha com validação de senha anterior.
  \item \textbf{deleteUser()}: Remove usuário e verifica existência.
  \item \textbf{validateLogin()}: Valida credenciais contra senha armazenada.
\end{itemize}

Todas as operações usam \texttt{BCryptPasswordEncoder} para tratamento de senhas.

\section{Camada de Apresentação}

Os controllers expõem endpoints HTTP e delegam lógica ao serviço:

\begin{itemize}
  \item \textbf{UserController}: 6 endpoints CRUD + busca.
  \item \textbf{AuthController}: 1 endpoint de validação de login.
\end{itemize}

Todos os endpoints usam anotações \texttt{@Operation}, \texttt{@ApiResponses} para documentação automática no Swagger.

\section{Tratamento de Exceções}

\texttt{GlobalExceptionHandler} centraliza o tratamento de exceções usando \texttt{@RestControllerAdvice}:

\begin{itemize}
  \item \textbf{UserNotFoundException}: HTTP 404 quando usuário não encontrado.
  \item \textbf{DuplicateEmailException}: HTTP 409 para email duplicado.
  \item \textbf{InvalidLoginException}: HTTP 401 para credenciais inválidas.
  \item \textbf{MethodArgumentNotValidException}: HTTP 400 para erros de validação.
\end{itemize}

Todas as respostas de erro seguem o padrão RFC 7807 (ProblemDetail).

A API adota versionamento explícito no path (\texttt{/api/v1}), permitindo evolução futura sem quebra de compatibilidade. Os endpoints foram projetados respeitando a semântica dos métodos HTTP, com separação clara entre atualização de dados e alteração de credenciais sensíveis.

\section{Justificativas de Design}

\subsection{Versionamento de API}

O versionamento via URI (\texttt{/api/v1}) facilita a coexistência de múltiplas versões e oferece clareza explícita ao cliente sobre a versão utilizada. Essa abordagem é amplamente adotada em APIs públicas consolidadas.

\subsection{Separação entre PUT e PATCH}

O método \texttt{PUT} é utilizado para atualização completa de dados do usuário (nome, email, login, role, endereço), enquanto \texttt{PATCH} é restrito à alteração de senha. Essa separação oferece segurança adicional e clareza semântica:

\begin{itemize}
  \item \textbf{PUT \texttt{/users/\{id\}}}: Atualiza múltiplos atributos não-sensíveis.
  \item \textbf{PATCH \texttt{/users/\{id\}/password}}: Modifica exclusivamente a credencial de acesso.
\end{itemize}

A utilização de \texttt{PATCH} para alteração de senha justifica-se por representar uma modificação parcial do recurso, alinhada às boas práticas de design REST.

\subsection{Busca com Filtros}

O endpoint \texttt{GET /api/v1/users?name=...} implementa busca por nome com processamento case-insensitive, permitindo buscas flexíveis sem distinção de maiúsculas/minúsculas.

\section{Tratamento de Erros}

Todos os endpoints implementam tratamento robusto de erros conforme RFC 7807 (Problem Details for HTTP APIs), retornando estruturas padronizadas de erro com informações de contexto:

\begin{lstlisting}[language=json, caption=Exemplo de Resposta de Erro (RFC 7807)]
{
  "type": "about:blank",
  "title": "Duplicate Email",
  "status": 409,
  "detail": "Email already registered",
  "instance": "/api/v1/users"
}
\end{lstlisting}

\chapter{Banco de Dados}

O sistema utiliza PostgreSQL 15 como banco de dados relacional, escolhido por sua robustez, confiabilidade comprovada e ampla adoção em ambientes corporativos e em nuvem.

\section{Estratégia de Segurança}

As senhas são armazenadas utilizando hash criptográfico com \textbf{BCrypt}, conforme recomendações da OWASP (Open Web Application Security Project). Essa abordagem oferece:

\begin{itemize}
  \item \textbf{Irreversibilidade}: Impossível recuperar senha original do hash.
  \item \textbf{Resistência a Rainbow Tables}: BCrypt utiliza salt aleatório por instância.
  \item \textbf{Adaptabilidade}: Fator de custo computacional pode ser ajustado ao longo do tempo.
  \item \textbf{Conformidade}: Alinhado com padrões de segurança da indústria.
\end{itemize}

\section{Integridade de Dados}

O banco de dados implementa restrições de integridade essenciais:

\begin{itemize}
  \item \textbf{Unicidade de Email}: Restrição UNIQUE na coluna \texttt{email}, prevenindo registros duplicados.
  \item \textbf{Unicidade de Login}: Restrição UNIQUE na coluna \texttt{login}, garantindo identificação única.
  \item \textbf{Validação de NOT NULL}: Campos obrigatórios protegidos em nível de banco.
  \item \textbf{Chaves Primárias}: Identificadores UUID garantem exclusividade global.
  \item \textbf{Auditoria Automática}: Campos \texttt{createdAt} e \texttt{updatedAt} mantêm histórico de modificações.
\end{itemize}

\section{Inicialização do Banco}

O arquivo \texttt{init.sql} contém scripts de inicialização que:

\begin{itemize}
  \item Criam esquema do banco de dados.
  \item Definem restrições e índices.
  \item Inserem dados de exemplo para testes.
\end{itemize}

A inicialização é executada automaticamente pelo Docker Compose durante o provisionamento da aplicação.

\chapter{Discussão}

\section{Análise das Decisões Arquiteturais}

As decisões arquiteturais adotadas demonstram um equilíbrio deliberado entre simplicidade operacional e robustez técnica. A Clean Architecture foi escolhida não por purismo teórico, mas por oferecer vantagens práticas mensuráveis:

\begin{itemize}
  \item \textbf{Testabilidade}: Isolamento de camadas permite testes unitários sem dependências de infraestrutura.
  \item \textbf{Manutenibilidade}: Separação de responsabilidades facilita localizável e correção de bugs.
  \item \textbf{Escalabilidade}: Estrutura em camadas permite evolução independente de cada componente.
  \item \textbf{Independência de Frameworks}: Lógica de negócio não é acoplada ao Spring Boot, facilitando migração futura.
\end{itemize}

\section{Trade-offs e Limitações}

O projeto reconhece as seguintes limitações do escopo atual:

\begin{itemize}
  \item \textbf{Ausência de Autenticação Baseada em Token}: Atualmente, não há suporte a JWT ou OAuth 2.0. A validação é stateless, porém sem geração de tokens. Essa funcionalidade pode ser incorporada em fases futuras sem impacto estrutural significativo.
  
  \item \textbf{Sem Autorização Granular}: Roles de usuário existem mas não há enforcement de autorização por endpoint. A implementação de `@PreAuthorize` pode ser adicionada facilmente à camada de controle.
  
  \item \textbf{Falta de Rate Limiting}: Sem proteção contra abuse ou DDoS na camada de API. Pode ser implementado via Spring Cloud Gateway ou bibliotecas especializadas.
  
  \item \textbf{Sem Auditoria Detalhada}: Não há registro de todas as ações (quem modificou, quando, de onde). Auditoria completa requer JPA Envers ou solução similar.
  
  \item \textbf{Ausência de Cache}: Sem caching distribuído (Redis) para melhorar performance em consultas repetidas.
\end{itemize}

\section{Potenciais Melhorias Futuras}

A arquitetura atual é preparada para incorporação de:

\begin{itemize}
  \item Autenticação e autorização com Spring Security + JWT.
  \item Cache distribuído com Redis.
  \item Event sourcing para auditoria completa.
  \item Implementação de CQRS (Command Query Responsibility Segregation).
  \item Documentação OpenAPI estendida com exemplos de erro.
  \item Monitoramento com Prometheus e Grafana.
  \item Service mesh (Istio) para orquestração avançada.
\end{itemize}

\section{Conformidade com Requisitos}

O projeto implementa 100\% dos requisitos funcionais e técnicos propostos:

\begin{itemize}
  \item \textbf{CRUD Completo}: 6 endpoints implementados (\texttt{POST}, \texttt{GET}, \texttt{PUT}, \texttt{DELETE}, \texttt{PATCH}).
  \item \textbf{Validação de Login}: Endpoint \texttt{POST /auth/validate} funcional.
  \item \textbf{Busca Avançada}: Suporte a filtros por nome com case-insensitive.
  \item \textbf{Segurança}: Senhas com BCrypt, validações em múltiplas camadas.
  \item \textbf{Documentação}: Swagger, Postman, README, ARCHITECTURE.
  \item \textbf{Testes}: 53 testes, 95.21\% cobertura.
  \item \textbf{Infraestrutura}: Docker Compose funcional e reprodutível.
\end{itemize}
