# RELATÓRIO TÉCNICO
## FIAP Tech Challenge - Fase 1: Sistema de Gestão de Usuários

---

## 1. INTRODUÇÃO

### 1.1 Contexto do Desafio

O FIAP Tech Challenge propõe o desenvolvimento de uma aplicação backend moderna para gestão de usuários em um sistema de marketplace de restaurantes. A **Fase 1** focou na implementação de um conjunto robusto de APIs RESTful com operações fundamentais de CRUD (Create, Read, Update, Delete) para entidade User.

### 1.2 Objetivo da Fase 1

Desenvolver uma aplicação **Spring Boot 3** com **Java 17 LTS** que atenda os seguintes requisitos funcionais e não-funcionais:

#### Funcionais
- ✅ Criar novo usuário com validação completa
- ✅ Atualizar dados de usuário (exceto senha)
- ✅ Alterar senha com validação de senha atual
- ✅ Deletar usuário
- ✅ Buscar usuário por ID
- ✅ Listar usuários por filtro de nome
- ✅ Validar credenciais (autenticação simples)

#### Não-Funcionais
- ✅ Código organizado em arquitetura em camadas
- ✅ Validação de dados robusta
- ✅ Tratamento de erros padronizado (RFC 7807 - ProblemDetail)
- ✅ Senhas armazenadas com BCrypt
- ✅ Documentação automática com Swagger UI (OpenAPI 3.0)
- ✅ Container Docker pronto para produção
- ✅ Testes de integração e E2E
- ✅ README e documentação completa

## 2. ARQUITETURA DA SOLUÇÃO

### 2.1 Visão Geral

A arquitetura segue o **padrão em camadas (Layered Architecture)**, garantindo separação clara de responsabilidades e facilidade de manutenção:

```
┌────────────────────────────────────────────────────────┐
│                  REST Controllers                      │
│      (UserController, AuthController)                 │
├────────────────────────────────────────────────────────┤
│              Business Logic (Services)                 │
│              (UserService)                            │
├────────────────────────────────────────────────────────┤
│            Data Access (Repositories)                  │
│           (UserRepository - JPA)                      │
├────────────────────────────────────────────────────────┤
│          JPA Entities (Domain Models)                  │
│      (User, Address, UserRole)                       │
├────────────────────────────────────────────────────────┤
│           PostgreSQL Database                         │
└────────────────────────────────────────────────────────┘
```

### 2.2 Justificativa das Decisões Técnicas

#### 2.2.1 Spring Boot 3.2.1
- **Motivo**: Framework mais moderno disponível
- **Vantagens**: 
  - Startup mais rápido (~3s vs 8s no Spring Boot 2)
  - Compatibilidade total com Java 17 records e outras features
  - Support até 2025 (LTS)
  - Spring Native ready para compilação GraalVM

#### 2.2.2 Java 17 LTS
- **Motivo**: Versão LTS com suporte até setembro 2026
- **Vantagens**:
  - Records (reduzem boilerplate)
  - Text blocks (melhor legibilidade)
  - Sealed classes (type safety)
  - Performance melhorada (~10% vs Java 11)

#### 2.2.3 PostgreSQL 15
- **Motivo**: Banco relacional maduro e open-source
- **Vantagens**:
  - JSONB nativo (extensibilidade futura)
  - Full-text search (buscas avançadas)
  - Constraint triggers (validações complexas)
  - ACID completo com MVCC

#### 2.2.4 Spring Data JPA com Hibernate
- **Motivo**: ORM mais consolidado do ecossistema Spring
- **Vantagens**:
  - Queries automáticas por nome de método
  - Lazy loading de relacionamentos
  - Caching automático de entidades
  - Suporte a múltiplos dialetos SQL

#### 2.2.5 BCrypt para Senhas
- **Motivo**: Hash adaptativo com salt aleatório
- **Vantagens**:
  - Implementação padrão de facto
  - Intrínseco suporte a cost factor (difficulty)
  - Não reversível (uma via)
  - Resistente a rainbow tables

#### 2.2.6 SpringDoc OpenAPI
- **Motivo**: Geração automática de documentação
- **Vantagens**:
  - Swagger UI integrado (sem dependências externas)
  - OpenAPI 3.0 spec completo
  - Atualização automática com código
  - Suporte a @Schema, @ParameterObject

## 3. MODELAGEM DE DADOS

### 3.1 Entidade User

#### Schema do Banco de Dados

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    login VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(50) NOT NULL DEFAULT 'CUSTOMER',
    address_street VARCHAR(255),
    address_number VARCHAR(10),
    address_complement VARCHAR(255),
    address_city VARCHAR(100),
    address_state VARCHAR(2),
    address_zip_code VARCHAR(10),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_modified_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX idx_user_email ON users(email);
CREATE UNIQUE INDEX idx_user_login ON users(login);
```

#### Atributos da Entidade

| Atributo | Tipo | Constraint | Regra de Negócio |
|----------|------|-----------|------------------|
| `id` | BigInt | PK, IDENTITY | Auto-incremento, não alterável |
| `name` | String(255) | NOT NULL | Nome completo do usuário |
| `email` | String(255) | NOT NULL, UNIQUE | Único no banco, validado com regex |
| `login` | String(255) | NOT NULL, UNIQUE | Credencial única para autenticação |
| `passwordHash` | String(255) | NOT NULL | Hash BCrypt, nunca senha plana |
| `role` | Enum | NOT NULL | RESTAURANT_OWNER ou CUSTOMER |
| `address` | Embeddable | NOT NULL | Endereço completo com 6 campos |
| `createdAt` | LocalDateTime | NOT NULL | Timestamp de criação (immutável) |
| `lastModifiedAt` | LocalDateTime | NOT NULL | Auto-atualizado em qualquer alteração |

#### Validações

```
name         → @NotBlank, comprimento > 3
email        → @Email + regex RFC 5322, UNIQUE
login        → @NotBlank, comprimento > 3, UNIQUE
password     → @NotBlank, comprimento >= 8
role         → @NotNull, valores pré-definidos
address      → @Valid, validação de cada campo
```

### 3.2 Objeto Embutido Address

Modelado como `@Embeddable` para representar endereço dentro de User:

```java
@Embeddable
public class Address {
    @NotBlank
    private String street;
    
    @NotBlank
    private String number;
    
    private String complement;
    
    @NotBlank
    private String city;
    
    @NotBlank
    @Size(min = 2, max = 2)
    private String state;
    
    @NotBlank
    private String zipCode;
}
```

### 3.3 Enum UserRole

Define os papéis de usuário permitidos:

```java
public enum UserRole {
    RESTAURANT_OWNER("Proprietário de Restaurante"),
    CUSTOMER("Cliente");
}
```

## 5. TRATAMENTO DE ERROS

### 5.1 Padrão RFC 7807 - ProblemDetail

Todas as respostas de erro seguem o padrão **RFC 7807** (Problem Details for HTTP APIs), garantindo consistência e facilidade de consumo:

```json
{
  "type": "https://api.fiap.com/errors/{status}",
  "title": "Título do Erro",
  "status": 400,
  "detail": "Descrição específica do erro",
  "instance": "/api/v1/endpoint/que/falhou",
  "timestamp": "2026-01-03T15:35:00Z",
  "traceId": "8c9e5f2a-1b3d-4e7f-9a2c-6d4f8e1a3b5c",
  "severity": "error",
  "error": "Bad Request",
  "errorCode": "UNIQUE_ERROR_CODE",
  "suggestion": "Sugestão para resolver o problema"
}
```

### 5.2 Códigos de Status HTTP

| Status | Situação | Exception Java |
|--------|----------|-----------------|
| **200** | OK - Sucesso em GET, POST válido | N/A |
| **201** | Created - Recurso criado com sucesso | N/A |
| **204** | No Content - DELETE, PATCH com sucesso | N/A |
| **400** | Bad Request - Validação de entrada falha | MethodArgumentNotValidException |
| **401** | Unauthorized - Autenticação falha | InvalidLoginException |
| **404** | Not Found - Recurso não existe | UserNotFoundException |
| **409** | Conflict - Email/Login duplicado | DuplicateEmailException |
| **500** | Internal Server Error - Erro inesperado | Exception genérica |

### 5.3 Custom Exceptions

#### UserNotFoundException
```java
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(String message) {
        super(message);
    }
}
```
- **Usado em**: GET /users/{id} quando ID não existe
- **Status**: 404
- **Exemplo**: "Usuário com ID 999 não encontrado"

#### DuplicateEmailException
```java
public class DuplicateEmailException extends RuntimeException {
    public DuplicateEmailException(String message) {
        super(message);
    }
}
```
- **Usado em**: POST /users, PUT /users/{id} quando email duplicado
- **Status**: 409
- **Exemplo**: "Email 'joao@example.com' já está registrado"

#### InvalidLoginException
```java
public class InvalidLoginException extends RuntimeException {
    public InvalidLoginException(String message) {
        super(message);
    }
}
```
- **Usado em**: POST /auth/validate quando login/senha inválidos
- **Status**: 401
- **Exemplo**: "Login ou senha incorretos"

### 5.4 GlobalExceptionHandler

Centraliza o tratamento de todas as exceções com @RestControllerAdvice:

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ProblemDetail> handleUserNotFound(
            UserNotFoundException ex, WebRequest request) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(buildProblemDetail(
                HttpStatus.NOT_FOUND,
                "Usuário Não Encontrado",
                ex.getMessage(),
                "error",
                "USER_NOT_FOUND",
                request
            ));
    }
    
    @ExceptionHandler(DuplicateEmailException.class)
    public ResponseEntity<ProblemDetail> handleDuplicateEmail(
            DuplicateEmailException ex, WebRequest request) {
        return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(buildProblemDetail(
                HttpStatus.CONFLICT,
                "E-mail Duplicado",
                ex.getMessage(),
                "error",
                "DUPLICATE_EMAIL",
                request
            ));
    }
    
    // ... outros handlers
}
```

## 6. INFRAESTRUTURA

### 6.1 Docker

#### Dockerfile - Construção da Imagem

O Dockerfile implementa **multi-stage build** para otimizar tamanho e segurança:

```dockerfile
# Stage 1: Maven builder
FROM maven:3.9.6-eclipse-temurin-17-alpine AS builder
WORKDIR /build
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests=true

# Stage 2: Runtime
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=builder /build/target/*.jar app.jar
RUN chown -R appuser:appgroup /app
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=10s --start-period=10s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-Xmx512m", "-Xms256m", "-XX:+UseG1GC", "-jar", "app.jar"]
```

**Características**:
- ✅ Multi-stage: reduz imagem de ~1.2GB para ~400MB
- ✅ Non-root user (segurança)
- ✅ Healthcheck integrado
- ✅ Otimizações JVM (G1GC, heap mínimo/máximo)
- ✅ Alpine Linux (imagem leve)

### 6.2 Docker Compose

Orquestra aplicação + PostgreSQL com configuração completa:

```yaml
version: '3.9'

services:
  postgres:
    image: postgres:15-alpine
    container_name: user-management-db
    environment:
      POSTGRES_DB: user_management
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - app-network

  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: user-management-api
    ports:
      - "8080:8080"
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/user_management
      SPRING_DATASOURCE_USERNAME: postgres
      SPRING_DATASOURCE_PASSWORD: postgres
    networks:
      - app-network

volumes:
  postgres_data:

networks:
  app-network:
    driver: bridge
```

**Características**:
- ✅ PostgreSQL com healthcheck
- ✅ Depends_on com condição de health
- ✅ Network bridge para comunicação entre containers
- ✅ Volume para persistência de dados

## 13. DECISÕES DE DESIGN

### 13.1 Segurança

**Por que não usar Spring Security?**
- Requisito: "validação simples em banco"
- Spring Security é framework completo para autenticação/autorização
- Implementação direta em UserService é mais leve e didática

**Por que BCrypt e não outro?**
- BCrypt é implementação padrão com salt automático
- Cada hash é diferente mesmo com mesma senha
- Adaptável: cost factor pode aumentar com evolução do hardware

### 13.2 DTOs

**Por que DTOs separados?**
- UserCreateRequest: permite validações diferentes
- UserUpdateRequest: omite senha propositalmente
- UserResponse: nunca expõe passwordHash
- Separação clara de responsabilidades

### 13.3 Address como Embeddable

**Por que não entidade separada?**
- Address é sempre vinculado a User (sem endereço "órfão")
- Simplifica queries (JOINs desnecessários)
- Reduz número de tabelas
